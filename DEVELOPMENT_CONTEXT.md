# DEVELOPMENT_CONTEXT.md

## このファイルの目的

このファイルは、「高速・下道コスパナビ」の開発背景・設計思想・判断基準を、Claude / Claude Code / ChatGPT などの相談相手や実装支援AIに引き継ぐための資料です。

`CLAUDE.md` は Claude Code に守らせる作業ルール、`PROJECT_HANDOFF.md` は直近の実装状況・引き継ぎ内容、この `DEVELOPMENT_CONTEXT.md` は長期的な設計思想・仕様判断の背景をまとめる役割です。

Claude Code に作業を依頼する際は、必ず最初に以下を読ませてください。

- `CLAUDE.md`
- `PROJECT_HANDOFF.md`
- `DEVELOPMENT_CONTEXT.md`

特に、Routes API 呼び出し数を増やさないこと、候補IC数上限を勝手に変更しないこと、大規模リファクタリングをしないことを最重要制約とします。

---

## アプリの目的

「高速・下道コスパナビ」は、現在地または任意の出発地から目的地までについて、以下を比較するWebアプリです。

- 全高速
- 途中ICで降りる
- 入口ICを変える
- 全下道

目的は、Google Maps のような完全なナビを作ることではありません。

主目的は、実車中または出発前に、

- 高速を使う価値があるか
- どの入口ICから乗るとよいか
- どの出口ICで降りるとコスパがよいか
- 高速料金に対して短縮時間が見合うか

を判断するための補助ツールを作ることです。

そのため、厳密な高速料金の完全一致よりも、「比較として自然か」「実用上の判断材料になるか」「APIコストが抑えられているか」を重視します。

## 最重要制約

このプロジェクトでは、以下を絶対に守る必要があります。

1. `config.js` / APIキーを触らない
2. Routes API 呼び出し数を増やさない
3. 候補IC数上限を勝手に変更しない
4. Google Maps Platform の無料枠・低コスト運用を重視する
5. `extraComputations:["TOLLS"]` を復活させない
6. `travelAdvisory.tollInfo` 前提の料金取得に戻さない
7. 旧関数・旧HTML・旧CSSを安易に削除しない
8. 大規模リファクタリングをしない
9. 小差分で進める
10. Puppeteer / Playwright / ヘッドレスChromeは禁止
11. 作業完了時は `CLAUDE.md` のルールどおり、最後に1回だけ `finish-check.cmd` を実行する

特に、Routes API 呼び出し数を増やす修正は原則禁止です。

候補ICを増やしたり、候補ごとに追加のRoutes APIを呼ぶような実装は、コスト増につながるため避けます。

## このアプリで重視する「正しさ」

このアプリで重視する正しさは、厳密なナビゲーションや高速料金の完全再現ではありません。

重視するのは以下です。

- 比較の方向性が自然である
- 高速を使うかどうかの判断材料になる
- 入口IC・出口ICの候補が実際の進行方向と大きくズレない
- ETC概算が極端に高すぎたり安すぎたりしない
- 首都高料金とNEXCO等の距離ベース料金が二重計上されない
- 実車中に見やすい表示である
- APIコストが増えない

多少の料金誤差は許容します。

ただし、以下のようなものは修正対象です。

- 明らかに料金が高すぎる
- 明らかに料金が安すぎる
- 料金がマイナスになる
- 節約額が不自然に大きい
- 本来候補に出るべきでないICが出る
- 進行方向と逆向きのICが候補になる
- 首都高分に距離ベース料金が二重計上される

## 料金計算の基本方針

Google Routes API の tollInfo は使わず、アプリ側の概算料金を使います。

料金計算は厳密な高速料金表の再現ではなく、判断材料としての概算です。

基本方針は以下です。

### 首都高

首都高は固定料金として扱います。

```js
SHUTO_TOLL_ESTIMATE_YEN = 1000
```

首都高区間に対して距離ベース料金を重ねると、料金の二重計上になりやすいため、首都高は原則として固定1,000円で扱います。

### NEXCO・京葉道路・東名・中央道など

首都高以外の高速道路・有料道路は、従来の距離ベース概算を使います。

厳密な料金表ではなく、距離に応じた概算です。

### 首都高と非首都高をまたぐ場合

現在のV2料金ルールは以下です。

- 首都高IC → 首都高IC  
  → 首都高固定1,000円のみ

- 首都高IC → 非首都高IC  
  → 首都高固定1,000円 + NEXCO入口以降の距離ベース概算

- 非首都高IC → 首都高IC  
  → 非首都高区間の距離ベース概算 + 首都高固定1,000円

- 非首都高IC → 非首都高IC  
  → 従来どおり距離ベース概算

このルールは、現在 `estimateComparisonCandidateToll(...)` に共通化されています。

対象は以下です。

- 通常検索
- 入口比較V2
- 出口比較V2

## 料金計算で避けたい問題

以前は通常検索側で、

```text
首都高固定1,000円 + startIc → endIc の全距離ベース概算
```

のようになり、首都高区間に対して距離ベース料金が二重計上される可能性がありました。

現在は、通常検索側の `estimateMainHighwayToll` も `estimateComparisonCandidateToll(...)` を経由するように統一済みです。

これにより、通常検索・入口比較V2・出口比較V2で首都高料金ルールが揃っています。

今後、料金が以前より下がるケースがありますが、それは首都高区間の二重計上を避けた結果であり、基本的には正しい方向です。

ただし、実ブラウザ確認で明らかに不自然な値が出る場合は調査対象です。

---

## 入口比較V2の考え方

入口比較V2は、現在地または出発地から、どの入口ICに向かうとよいかを比較する機能です。

基本構成は以下です。

```text
現在地/出発地
→ 候補入口ICまで下道
→ 高速区間
→ 目的地側ICから目的地まで下道
```

入口候補は、実際の進行方向や現在地からの使いやすさを重視します。

特に首都高入口については、現在地から最も近い候補を優先しつつ、進行方向上の本線IC候補と組み合わせます。

表示では、候補ごとに以下を出します。

- あと何分で入口に着くか
- 何分短縮できるか
- ETC概算
- 料金目安
- 円/分
- 合計時間

入口カードには、料金内訳として以下のような表示を入れています。

```text
料金目安：首都高 約1,000円 + 他道路 約165円
```

非首都高候補の場合は、以下のような表示になります。

```text
料金目安：他道路 約165円
```

この内訳表示は、追加のRoutes API呼び出しを行わず、

```text
estimatedToll - shutoToll
```

で他道路分を逆算しています。

## 出口比較V2の考え方

出口比較V2は、全高速で目的地側まで行く場合と、途中ICで降りる場合を比較する機能です。

目的は、少し遅くなっても途中で降りることで料金節約になるかを判断することです。

表示では、候補ごとに以下を出します。

- どの出口で降りるか
- 何円節約できるか
- 何分遅いか
- 円/分
- 料金目安

出口比較カードには、以下のような料金目安表示を入れています。

```text
料金目安：通常 約1,431円 / この出口 約1,000円
```

ここで「通常」は、通常検索側の料金計算と整合する必要があります。

通常検索側の料金ロジックもV2ルールに統一済みのため、今後は出口比較V2の「通常 約◯円」とトップパネルの通常料金が自然に整合することが望ましいです。

## IC候補選定の考え方

候補IC選定では、以下を重視します。

- 実際の進行方向に沿っている
- 現在地から使いやすい
- 候補数を増やしすぎない
- API呼び出し数を増やさない
- JCTや選択不可ICを出さない
- 目的地と道路方面が大きくズレない

候補ICは、原則として進行方向順に並べます。

首都高入口候補は、現在地側の最寄り候補を優先する方針です。

通常モードと実車テストモードで候補数上限がありますが、これを勝手に変更してはいけません。

---

## 方面判定の考え方

方面判定は、過去にキーワードベース・目的地名ベースの判定を検討しましたが、地名や施設名によって誤判定が起きやすいため、現在は距離やルート上のIC候補を重視する方針です。

現在地から目的地までのルート上にある、これから使用予定のインターチェンジを距離ベースで選ぶ方式を重視しています。

キーワードで無理に道路方面を決めるより、ルート形状・距離・進行方向に基づいて候補を絞る方が望ましいです。

---

## 首都高の扱い

首都高はこのアプリの中で特別扱いです。

理由は、首都高が都市高速であり、NEXCO等の距離ベース概算と同じ扱いにすると不自然な料金になりやすいためです。

首都高は固定1,000円で扱い、非首都高区間とは分けて料金を計算します。

首都高入口候補は、現在地からの使いやすさが重要です。

例：荒川区役所付近から千葉方面へ向かう場合、堤通・上野・小松川・篠崎などが絡むことがありますが、料金計算上は「首都高区間」と「京葉道路/NEXCO相当区間」を混同しないことが重要です。

---

## endIc / exitIc 側の注意点

料金ロジック統一後、今後チェックすべきポイントとして、終点側ICの選択があります。

入口側は首都高入口やNEXCO入口の扱いを整理済みですが、終点側では以下のようなズレが起きる可能性があります。

- 本来のNEXCO出口ではなく、首都高出口を拾ってしまう
- 目的地に近いだけの別ICを拾ってしまう
- 本来の出口より手前または先のICを選んでしまう
- 通常検索の終点ICと出口比較V2の候補出口ICの扱いがズレる

この場合、以下のような不自然な表示につながります。

- 通常検索のETCが高すぎる
- 通常検索のETCが安すぎる
- 出口比較の節約額が大きすぎる
- 出口比較の節約額が0円またはマイナスになる
- 「通常 約◯円 / この出口 約◯円」が直感と合わない

ただし、現時点で必ずバグがあるという意味ではありません。

まずは実ブラウザで複数目的地を確認し、不自然な結果が出た方面から調査します。

## APIコスト方針

このアプリは、Google Maps Platform の無料枠・低コスト運用を重視しています。

目標は、個人利用として月数百円程度、できれば月500円以内の運用です。

そのため、以下は禁止・要注意です。

- 候補ICごとにRoutes APIを追加で呼ぶ
- 候補数を増やす
- 自動更新頻度を上げる
- `extraComputations:["TOLLS"]` を復活させる
- Routes API のEnterprise相当機能を使う
- 実車中に高頻度で自動再検索する

現在の方針では、Routes API: Compute Routes Pro を中心に使い、料金はアプリ側の概算で補います。

## UI表示の考え方

実車中または出発前に判断しやすいことを重視します。

重要なのは、細かい情報を大量に出すことではなく、

- どれがおすすめか
- 何分短縮できるか
- いくらかかるか
- 1分短縮あたり何円か
- 途中で降りると何円節約できるか
- その代わり何分遅くなるか

がすぐ分かることです。

表示が長くなりすぎた場合は、情報の削除よりも、フォントサイズ・色・行間・表示順の整理を優先します。

料金内訳表示は有用ですが、カードが長くなりすぎる場合は、弱い色や小さめフォントで調整します。

## 色表示の考え方

基本的に、時間比較では短い方を緑、長い方を赤にする方針です。

高速利用と有料回避の2値比較では、常に以下を基本にします。

- 短い方：緑
- 長い方：赤
- 片方なし：薄いグレー
- 同等：専用色を検討

色は、直感的に分かりやすいことを重視します。

## 自動更新の考え方

実車中の自動更新は便利ですが、APIコスト増につながるため慎重に扱います。

現在は、自動更新初期OFFの方針です。

自動更新条件は、移動距離や経過時間によって制御します。

出発地欄に文字が入っている場合は、GPS現在地ではなく入力出発地を使う意図があるため、自動更新との整合に注意します。

## バグ調査時の見方

料金や候補ICが不自然な場合、まず以下を疑います。

### 料金が高すぎる場合

- 首都高区間に距離ベース料金が二重計上されていないか
- fallbackDistanceMeters が全体距離になっていないか
- endIc が実際より遠いICになっていないか
- NEXCO区間の距離推定が過大ではないか

### 料金が安すぎる場合

- NEXCO出口を拾えていないのではないか
- 首都高固定1,000円だけになっていないか
- 非首都高区間の距離ベース概算が0円扱いになっていないか

### 節約額が不自然な場合

- 通常検索側の料金が過大または過小ではないか
- 候補出口側の料金が過大または過小ではないか
- 通常側と候補側で料金計算ルールがズレていないか

### 候補ICが不自然な場合

- 進行方向順になっているか
- JCTや選択不可ICが混ざっていないか
- routeBranch / branchOrder が正しく効いているか
- 首都高IC判定 `isShutoIc` が不足していないか
- googleName がGoogle Maps側と合っているか

### GoogleのtollInfoに戻すこと

Routes API の tollInfo を使うとAPIコストやSKUの問題が出るため、現在は使わない方針です。

動いている部分を壊しやすいため禁止です。

必要な場合も、まず小さな関数追加・小差分修正で進めます。

## パッチ的修正よりも統一パイプライン接続を優先する方針（2026-07-22追記）

このアプリでは、同じ種類の課題（道のり上の座標をどう扱うか、ICがどの道路に属するかをどう判定するか等）に対して、過去のセッションで複数の別々の関数・パイプラインが、その場その場の目的のためだけに作られてしまう傾向があった。結果として、同じルート・同じ数値が、実際にはどの関数を経由して計算されたものか追いにくくなり、バグ調査のたびに「表示と計算で参照先が違うのでは」といった誤解が生じやすくなっていた（既知の保留事項24の調査過程を参照）。

今後は、以下の方針で進める。

- 新しい課題（バグ修正・精度向上）に着手する際、まず「この課題を解決できる、既存の統一的な仕組み・パイプラインが既にこのアプリの中にないか」を確認する。特に、IC境界検出パイプライン（Step1〜7、`detectIcsNearPolyline`〜`buildTollCategorySequenceWithGapDetection`）や、そこで使われている密なサンプリング座標（`sampledPoints`）は、道のり上の座標精度が必要になる場面で繰り返し再利用できる可能性が高い基盤である
- 「まず狭い範囲だけのパッチで目先の症状を止め、既存の統一パイプラインへの接続は後回しにして改めて検討する」という進め方は、原則として避ける。範囲が広がってでも、既存の統一的な仕組みに接続する方向を先に検討し、それが難しい場合にのみ、理由を明記した上でパッチ的な対応に留める
- これは「小差分で進める」というCLAUDE.mdの基本方針と矛盾しない。統一パイプラインへの接続作業自体を、必要であれば複数の小さなステップに分割して進めればよい。「作業を小さく分ける」ことと「対症療法的な部分修正で済ませる」ことは別の話であり、後者を避けるために前者を活用する

## 今後の優先順位

現時点では、コード修正に入る前に、複数目的地での実ブラウザ確認を優先します。

特に見るべきポイントは以下です。

- 通常検索のETC概算が自然か
- トップパネルの料金内訳が自然か
- 入口比較の「首都高 + 他道路」内訳が自然か
- 出口比較の「通常 約◯円 / この出口 約◯円」が自然か
- 節約額がマイナスや不自然な値になっていないか
- Routes API回数が増えていないか

そのうえで、不自然な方面が見つかったら、以下を優先的に調査します。

1. `endIc / exitIc` 側の選択
2. `polylineAnalysis.nexcoEntranceIc / nexcoExitIc` の取得状況
3. `isShutoIc` 判定不足
4. `fallbackDistanceMeters` の過大利用
5. 候補ICの `googleName` のズレ

## まとめ

このプロジェクトでは、完璧な高速料金再現よりも、

- 低APIコスト
- 実車中の判断しやすさ
- 自然な候補IC
- 自然な料金概算
- 小差分で安全に進めること

を重視します。

Claude / Claude Code は、既存仕様を尊重し、API呼び出し数を増やさず、候補IC数上限を変えず、小さな修正を積み重ねる方針で作業してください。
